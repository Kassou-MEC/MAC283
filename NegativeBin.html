Number Systems · Lecture 04

How computers store
negative numbers

A processor has no minus sign. It has only wires that are on (1) or off (0). So how does it represent −5? This lecture builds the answer step by step — from the naïve idea to two's complement, the method real CPUs use — then gives you 10 problems to practice.

Why this is hard

A register is just a row of switches

Below is one 8-bit register holding the number 5. There is no slot for a “−” symbol. Every bit is already doing a job. To store −5, we must agree on a code — a rule that turns some pattern of 0s and 1s into a negative value.

8 bits → each column is a power of two. Add the lit columns: 4 + 1 = 5

The hardware can only do two things to a bit: leave it 0 or make it 1. Any meaning beyond that — like “this number is negative” — has to be a convention we invent.
The convention must also make arithmetic work. We want the CPU to add −5 the same way it adds +5, using one adder circuit. That requirement quietly decides the winner.
Idea 1

Sign-Magnitude — the obvious guess

Reserve the leftmost bit as the sign: 0 means +, 1 means −. The other 7 bits hold the plain magnitude. Click the sign bit to flip between +5 and −5.

Value: +5

(amber-outlined bit = sign · click it)

Why it’s tempting

  • Reads like how humans write numbers
  • Easy to convert by eye

Why CPUs reject it

  • Two zeros: 00000000 = +0 and 10000000 = −0. Wasteful and confusing.
  • Hard math: adding requires checking signs first — extra circuitry, slower.
Idea 2

One's Complement — flip every bit

To negate a number, invert all its bits (every 0→1 and 1→0). Type a small positive number and watch the flip produce its negative form.

flip every bit ↓
Progress: arithmetic is now much easier than sign-magnitude. But we still have two zeros — 00000000 (+0) flips to 11111111 (−0). One annoying flaw remains. The fix is tiny.
Idea 3 ★ · the standard

Two's Complement = flip, then add 1

One extra step erases the double-zero problem and makes addition “just work”. This is what virtually every modern CPU uses. The recipe to convert a negative number:

1

Write the magnitude

Take the positive version of the number in plain binary, padded to your bit width.

2

Flip all bits

Invert every digit (the one's complement): 0→1, 1→0.

3

Add 1

Add binary 1 to the result. Done — that pattern is the negative number.

One zero only. +0 and the “flip+1” of 0 both come out to 00000000 (the overflow bit falls off the left). No more −0.
Read the sign instantly: the leftmost bit is still 0 for positives and 1 for negatives — for free, no special rule needed.

Speed technique

The exam shortcut — no addition needed

There's a faster way that skips the “add 1” carry step. Scan the magnitude from the right:

  • Copy bits unchanged up to and including the first 1 you meet.
  • Flip everything after that first 1.

green = copied as-is · amber = the pivot (first 1) · blue = flipped

−5 again: magnitude 00000101. From the right, copy 1 (the first 1), then flip the rest → 11111011. Same answer, zero arithmetic.
Limits & overflow

How far can n bits reach?

In two's complement, the negative side reaches one step further than the positive side, because the single zero lives on the positive half.

The formula: for n bits the range is −2ⁿ⁻¹ to 2ⁿ⁻¹ − 1.
Go past the limit and you get overflow — the value silently wraps around. A classic CompArch bug source.

One-page cheat sheet

The recipe (negative → binary)

  • Write the magnitude in binary, padded to width
  • Flip every bit (one's complement)
  • Add 1 → that's the two's complement
Shortcut: copy from the right through the first 1, flip the rest.

Things to remember

  • Leftmost bit = sign (0 = +, 1 = −)
  • Two's complement has one zero
  • Range for n bits: −2ⁿ⁻¹ … 2ⁿ⁻¹−1
  • To go back: flip + add 1 again returns the magnitude
