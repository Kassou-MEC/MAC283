Definition

What is Combinational Logic?

Combinational logic is a digital circuit whose output depends only on the current inputs — never on past inputs. There is no memory and no stored state.

The one-line test

Change an input and the output updates immediately. If you cover the inputs, the circuit remembers nothing about what came before. That "no memory" property is the whole idea — and it's what separates combinational logic from the sequential logic (flip-flops, registers) coming later in the course.

Contrast: a light switch is combinational (position now = light now). A counter that remembers how many times you pressed it is not — it has memory, so it's sequential.

Why does "no memory" matter so much? Because it makes the circuit predictable and instant. Give it the same inputs and you always get the same output, every time, with no hidden history to trip you up. That reliability is exactly what you want for arithmetic and decision-making inside a processor — the ALU must give 0011 + 0101 = 1000 the same way on the first try and the millionth.

🌍 Real-world example — a doorbell

Press the button and it rings now; let go and it stops. It doesn't count how many times you rang or remember you were here yesterday. Output (ringing) depends only on the input (button pressed) at this instant — a perfect everyday picture of combinational logic.

Compare a parking-garage counter that tracks how many cars have entered: it must remember the running total, so it needs memory — that's sequential logic, coming later in the course.

The Rules

Characteristics & everyday examples

Four defining traits

  • No memory elements — nothing is stored
  • No feedback — signals flow forward, no loops
  • Instant output for a given input
  • Built from logic gates

Examples: adders, multiplexers, encoders.

Compare to real life

💡 Light switch — flip it and the light is ON or OFF instantly. It doesn't remember yesterday, only what the switch is doing now.

🔢 Calculator button — press 3 + 5 = and logic circuits (adders) compute right away. It doesn't care what you pressed before.

Each trait has a practical reason: no memory keeps the answer honest to the current inputs; no feedback loops means signals never chase their own tail, so the output settles in a fixed, tiny amount of time; instant output is what lets a CPU do billions of these per second; and being built only from gates means anything here can be drawn as a wiring diagram of AND/OR/NOT.

🌍 Real-world example — a microwave keypad

Punch in 90 and the display shows 90 the instant you press each digit — the decoder driving the screen reacts immediately to whichever key is down, with no feedback and no memory of the last thing you cooked. The timer that counts down afterward is a different animal (it must remember the remaining seconds) — that part is sequential.

A logic gate is a tiny decision maker: it takes one or two yes/no signals (1 = on, 0 = off) and produces one yes/no answer by a fixed rule. Every gate is its truth table — the complete list of "for these inputs, this output." Big Picture

From gates to building blocks

Think of logic gates as LEGO bricks, and logic blocks as the buildings you make from them. Gates do basic Boolean operations; blocks combine many gates to do higher-level jobs.

ConceptLogic GateLogic Block
RoleBasic Boolean opHigher-level task
ExamplesAND, OR, NOT, XORMUX, Decoder, Adder
RelationshipFoundation unitsCombinations of gates

Common combinational blocks

  • MUX — selects one input from many
  • DEMUX — routes one input to many outputs
  • Encoder — 2ⁿ inputs → n-bit code
  • Decoder — n-bit input → 2ⁿ output lines
  • Adders / Subtractors — binary arithmetic
  • Multipliers — fast multiply in CPUs/GPUs
🌍 Real-world example — building a house

Bricks (gates) are simple and identical. Combine them one way and you get a wall; another way, a staircase — those are your logic blocks (MUX, decoder, adder). Combine blocks and you get a whole house: the ALU, then the CPU, then your phone. Same handful of brick types the whole way down; the cleverness is in the arrangement.

Interactive Block

MUX — the input selector

A 4-to-1 multiplexer picks one of four data lines to pass through, chosen by the two select bits S1 S0. Set the inputs and the selector, and watch which line reaches the output.

The rule is simple counting in binary: the select bits name a line number. 00→I0, 01→I1, 10→I2, 11→I3. With n select lines you can choose among 2ⁿ inputs, so 2 selects handle 4 inputs, 3 selects handle 8. A MUX is how a chip shares one wire (the output) among many possible sources — a traffic cop waving through one lane at a time.

🌍 One more — a restaurant's single pickup window

Four cooks (I0–I3) finish plates, but there's one pickup window (the output). The expediter's ticket number (the select lines) decides whose plate goes out next. Only one passes at a time; the rest wait. Inside a CPU, a MUX does this to choose which register's value flows into the ALU.

Interactive Block

DEMUX — the router

The opposite of a MUX: one input is routed to one of four outputs, chosen by S1 S0. Only the selected line carries the input value; the rest stay 0.

If a MUX is "many-in, one-out," a DEMUX is "one-in, many-out." The select bits again name a line number, but this time they decide the destination. Everywhere else stays 0, which matters: it means the input signal is delivered to exactly one place and nowhere else — no crosstalk, no leaks.

🌍 One more — a railroad switch yard

One incoming track (the input) carries a train, and a switch lever (the select lines) sends it down exactly one of several branch lines. Flip the lever and the same train heads to a different platform. A DEMUX is that switch for electrical signals — and it's how a CPU decides which register receives a computed result.

Interactive Block

Encoder — many lines to a code

An 8-to-3 encoder takes 8 input lines (one active at a time) and outputs the 3-bit binary code of which line is active. Click an input line.

Encoders compress "which one of many?" into a compact number. 8 lines need only 3 bits because 2³ = 8; 16 lines would need 4 bits. Instead of running 8 separate wires across a chip, you run 3 and send the code — far fewer wires, same information. That's why encoders sit wherever many buttons or sensors must report to a processor.

🌍 One more — an elevator call panel

Eight floor buttons, but the controller doesn't want eight wires — it wants a floor number. Press "floor 5" and the encoder outputs 101. One button in, a tidy 3-bit code out. The same trick turns a game controller's buttons into the compact codes your console reads.

Interactive Block

Decoder — a code to one line

The reverse of an encoder: a 2-to-4 decoder reads a 2-bit code A1 A0 and activates exactly one of its four output lines.

A decoder expands a compact code back into "point at exactly one thing." n input bits light up 1 of 2ⁿ outputs — 2 bits pick 1 of 4, 3 bits pick 1 of 8. This is how a processor selects one memory cell out of billions: the address bits feed a giant decoder that enables only the addressed location.

A seven-segment decoder takes the 4-bit code for a digit and lights exactly the right bars to draw it — code for "8" lights all seven segments, code for "1" lights just two. One small number in, one specific pattern out. Every glowing number on an appliance display is a decoder at work.

Interactive Block

Half adder & full adder

A half adder adds 2 bits. A full adder adds 3 bits — including a carry-in from the previous column, which is what lets us chain them into wide adders. Flip A, B, and Cin below.

Why is the carry-in the whole point? Because real numbers have many columns. When you add 1 + 1 you get 10 — a 0 with a carry into the next column, exactly like 9 + 1 = 10 in decimal. A half adder can't accept a carry from the right, so it can only ever do the rightmost column. A full adder can, so you can stack them to add 4-bit, 8-bit, 64-bit numbers.

🌍 Real-world example — grade-school column addition

You already run a "ripple carry" by hand. Adding 27 + 15, you do the ones column (7+5=12, write 2, carry 1), then the tens column with that carried 1 (2+1+1=4) → 42. Each column is a full adder: two digits plus the carry coming in from the right. Binary works identically — just with 0s and 1s and the rule 1+1=10.

Interactive · The Core Example

Ripple-carry adder — add it bit by bit

Chain four full adders and the carry ripples left. Enter two 4-bit numbers (0–15) and step through each column, exactly as the ALU does. Try the classic 3 + 5 first.

"Ripple" describes the timing: bit 0 must finish before bit 1 knows its carry-in, which must finish before bit 2, and so on — the carry travels left like a wave. It's simple and cheap, but the wider the number, the longer the ripple takes; that speed limit is why fast CPUs use cleverer "carry-lookahead" adders later on. Same math, less waiting.

🌍 Real-world example — a bucket brigade

Picture people in a line passing water buckets. Each person (a full adder) can only pass their bucket (the carry) after receiving one from the right. The water "ripples" down the line — and the longer the line, the longer the last bucket takes to arrive. Widen the adder from 4 bits to 64 and you've simply added more people to the brigade.

Block

Multipliers — fast by design

Multiplying in binary is just shift-and-add: each 1-bit in one number copies the other number, shifted into place, and all those partial products are summed. A hardware multiplier does every shift at once with a grid of AND gates (to form partial products) feeding a tree of adders (to total them) — so a multiply that would take many addition steps happens in a single pass.

A multiplier performs binary multiplication far faster than repeated addition. It's built from arrays of AND gates and adders — the AND gates form partial products, the adders sum them.

Used heavily in DSP, AI math, and graphics.

Where you feel it

A GPU rotating, scaling, and rendering 3D objects runs multipliers constantly for matrix and vector math — thousands of them in parallel, every frame.

Pattern to notice: even a "multiply" block is really just gates + adders arranged cleverly. Blocks are always made of simpler blocks and gates.
🌍 Real-world example — an Instagram filter

When you brighten or blur a photo, every pixel's color is multiplied by a number, millions of times per image. On video (30+ frames a second) that's billions of multiplications — which is exactly why phones and GPUs pack thousands of hardware multipliers running in parallel. The same silicon powers game graphics and AI models.

Putting It Together

Inside the CPU: 3 + 5

You type 3 + 5. Step through the pipeline and watch which combinational block does each job.

This is the whole lecture in one journey. No single block is smart — the encoder just makes a code, the decoder just enables a unit, the MUX just picks inputs, the adder just sums, the DEMUX just routes. Chained together, these dumb little combinational blocks add two numbers and show the answer. Intelligence lives in the arrangement, not any one part.

🌍 Real-world example — a factory assembly line

Raw parts enter one end; each station does one small job and hands off to the next, until a finished product rolls out. Your keystroke is the raw part, each combinational block is a station, and "3 + 5 = 8 on screen" is the finished product — assembled in billionths of a second.

How Deep It Goes

How many gates to add 3 + 5?

Each full adder is built from 2 XOR + 3 AND + 2 OR gates. To add two 4-bit numbers you chain N full adders. Change N and count.

28 gates for a 4-bit add sounds like a lot — until you scale it. A 64-bit adder is ~448 gates, and that's one operation. A modern CPU holds tens of billions of transistors (gates are built from transistors), which is how a chip the size of a fingernail runs an entire adder, multiplier, decoders, and more, all at once. Small rules, staggering scale.

Total = 28 logic gates
Every keystroke you make fires hundreds of these arithmetic and logic operations per second — all of it built from a handful of gate types.
🌍 Real-world example — bricks in a skyscraper

No single brick is impressive, but stack millions with intent and you get the Empire State Building. Gates are those bricks: seven simple types, repeated billions of times, become a processor that streams video and runs AI. When you feel your laptop is "powerful," you're really feeling scale plus arrangement.

Committed to Memory

Combinational logic — the recap

  • Output depends only on current inputs — no memory
  • Built from gates: AND, OR, NOT, NAND, NOR, XOR, XNOR
  • MUX selects one of many; DEMUX routes to one of many
  • Encoder: 2ⁿ→n bits · Decoder: n bits→2ⁿ lines
  • Full adders chain into ripple-carry adders (the ALU)

The takeaway

When you write Python or C, it becomes instructions like ADD R1,R2,R3 — and everything that runs it is logic gates combined into logic blocks. You're not just learning theory; you're learning the language of the machine.
